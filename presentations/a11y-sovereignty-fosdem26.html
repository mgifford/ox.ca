Accessible Sovereignty: Why the Four Freedoms Depend on Inclusion

Sovereignty is a practice; accessibility is the power to exercise it.

Mike Gifford, CivicActions

FOSDEM 2026 | Digital Sovereignty Track
Date & Location TBD

Opening: Sovereignty is a discipline, not a product. So is inclusion. If you stop practicing, it atrophies. Inclusion operationalizes freedom. I’m here to talk about a 'bug' in the way we view freedom. A legal right (the license) without functional power (accessibility) is an empty promise. Full submitted abstract is preserved in the comment above the cover slide.

The Core Premise

  • The foundation: Digital sovereignty is built on the Four Freedoms
  • The problem: Digital freedom is an illusion without accessibility
  • The thesis: Accessibility is the infrastructure that makes software freedom possible
  • Sovereignty = control; reliance on intermediaries erodes autonomy
  • If you cannot perceive/operate the tool, you are not sovereign
Define digital sovereignty as the ability for people and their communities to control and adapt the digital tools, data, and infrastructure they rely on. You have no freedom without accessibility: “open without accessible” is exclusion — a library with the lights turned off vs a living, visible forest where everyone can use, study, and share. Pair governance with tooling: just as DejaCode keeps licensing honest, ship accessibility automation (e.g., axe) in CI/CD so freedom is testable, not aspirational.

Re-evaluating the Four Freedoms

  • Freedom 0 (Run): Requires perceivable, operable UI
  • Freedom 1 (Study): Requires accessible documentation, open standards, and accessible CLIs
  • Freedom 2 (Redistribute): Requires accessible forges (GitHub, GitLab) and accessible package managers
  • Freedom 3 (Modify & share): Requires accessible editors/IDEs and accessible authoring tools (ATAG 2.0)
Map 0–3 to POUR: P/O unlock run; P/U unlock study; R keeps outputs and forks usable across contexts. Freedom 2: forges must be navigable and usable. Freedom 3: editors and IDEs (VSCode, Vim, etc.) and authoring platforms (ATAG 2.0). Cite FSF Four Freedoms: https://www.gnu.org/philosophy/free-sw.en.html#four-freedoms. If POUR breaks, the freedoms are theoretical.

Accessibility: The Foreground of Sovereignty

  • Accessibility long predates free software: a tradition of adaptive tech
  • Digital accessibility is the everyday practice of sovereignty
  • Think beyond “me-now”: design for future-you and for others
  • Accessibility and free software both expand options over time
  • Build for variation and resilience; preserve paths to participation
Frame accessibility as the practical foreground of sovereignty—functional power to exercise rights. Barriers to access existed long before FOSS; see historical inventions: https://cs.stanford.edu/people/eroberts/courses/soco/projects/2005-06/accessibility/firstwave.html. Both accessibility and free software ask us to think bigger than the present self, giving the future “me” more options. Use POUR as the operational lens that keeps freedoms real across contexts.

Community Keeps Freedom Alive

  • FOSS thrives because people care enough to fix and share
  • Accessibility thrives when communities care about present and future selves
  • If you cannot participate fully, you cannot help shape the software
  • Inclusive design covers permanent, temporary, and situational barriers
  • Personalization is a feature, not a fork: build for variation in ability
Replace the Machine/Mycelium metaphor with community reality. Microsoft Inclusive Design lens: permanent, temporary, situational barriers all need coverage. Accessibility is the same adaptability that lets FOSS communities riff and personalize for today’s needs.

Inclusive Design: The Multiplier

  • Permanent: low vision, mobility limits, cognitive load
  • Temporary: broken arm, eye infection, jet lag
  • Situational: glare, bad Wi‑Fi, noisy room
  • Curb-Cut Effect: screen reader fixes often improve SEO and performance
  • Design for variation; better defaults for everyone
These states rotate through a community. Building flexibility now protects the future-you and keeps the Four Freedoms usable when circumstances change.

Participant Engagement

  • Show of hands: Is software freedom a human right?
  • Follow-up: Have you audited your own contribution workflow for accessibility this year?
  • Gap reality: Most hands go down — that is the sovereignty gap
Use this early to align values vs practice. Prompt attention to pipelines: PR templates, issue forms, CI logs, CLI prompts.

The Sovereignty Gap

  • When a person cannot report a bug or suggest a fix, Freedom 3 is lost
  • Exclusion from contribution turns sovereign users into passive consumers
  • Fix the gap: accessible forges, forms, CI, and review checklists
Tie back to POUR and governance: contribution workflows must be perceivable, operable, understandable, and robust. Link the handout table for role-based actions.

Industry Momentum: The Accessible Forge

  • Infrastructure providers set the floor: accessible repos, CI, and design systems
  • Example: GitHub’s accessibility pledge and Primer accessibility work
  • Automate checks: axe in CI/CD, alongside license/compliance tools (e.g., DejaCode)
  • Outcome: when the forge is accessible, downstream ecosystems widen
Link to GitHub commitments: https://accessibility.github.com/ and https://github.blog/open-source/social-impact/our-pledge-to-help-improve-the-accessibility-of-open-source-software-at-scale/. Pair accessibility linting with SCA/license checks so pipelines guard freedom and compliance together.

CMS Governance for Accessibility

  • Ship accessible defaults and pattern libraries by default
  • Require accessibility checks in PR/release workflows
  • Make upstream-first fixes part of the contract, not a favor
  • Assign an accessibility owner in project governance
Ground the abstract in CMS practice: Drupal/WordPress can bake accessibility into governance, review checklists, and pattern libraries. Upstream-first fixes prevent forks of inaccessible code. An explicit accessibility role in release governance keeps the discipline funded and accountable.

The Right to Exist in the Code

POUR is the operating system for the Four Freedoms.

  • Freedom 0 (Run): Requires perceivable, operable UI
  • Freedom 1 (Study): Requires open documentation and open standards
  • Ethical failure: Inaccessible docs redact the right to learn
Map WCAG/POUR to functional rights: P/O unlock run; P/U unlock study; R keeps modified versions usable across exports/APIs. Without POUR, the freedoms are theoretical.

The Issue Queue: Our Forest Clearing

  • Collaboration is the engine of sovereignty
  • Open issue queues must be accessible to enable participation
  • An inaccessible tracker is a locked microphone in the town hall
  • Pick forges (GitHub, GitLab, Drupal.org) that prioritize the accessible forge
Participation gates define who holds power. Make contribution pipelines inclusive.

Biodiversity = Robustness

  • Maintainers: the deep roots
  • Lived-experience contributors: the mycelial connectors
  • Service firms: the connective tissue (Cygnus model)
  • Policy-makers (OSPOs/ZenDiS): the forest rangers
  • Investors: the catalytic nutrients
  • Sustainability advocates: the sunlight
Six-species model: diversity keeps the ecosystem resilient; accessibility invites missing species.

Accessibility & Sustainability

  • Lightweight code is green and accessible
  • Emergencies demand tools that work for everyone
  • Transparency: do not hide barriers; report them publicly
Cite Digital Accessibility Ethics; align eco-efficiency with inclusion.

Measuring What Matters (Sovereignty KPIs)

  • Upstream %: fix the core, not the fork
  • A11y patch latency: treat access barriers like security bugs
  • ARRM model: shared responsibility across roles
  • Public Money → Accessible Public Code: accessibility mandated in procurement
KPIs convert ethics to operations; procurement enforces inclusion.

Fund the Mycelium

  • Write better procurement
  • Publish your upstream policy
  • Hire lived-experience developers
  • Ship an accessibility defaults module
  • Require accessibility sign-off in PR templates
  • Budget for upstream accessibility fixes
Closing: Sovereignty is practiced. Fund the roots, or the forest dies. End with action: money, policy, hiring. Inclusion is the practice that sustains freedom.

Freedom Fails Without Accessibility

  • Run = Perceivable & Operable
  • Study = Perceivable & Understandable
  • Redistribute = Robust outputs across contexts
  • Modify = Operable tools + Robust, standards-based forks
One-slide summary linking the Four Freedoms to POUR. If accessibility breaks, sovereignty is hollow. Handout: see presentations/handouts/accessible-sovereignty-handout.html for FSF ↔ Accessibility mapping and role owners.

A New Definition of Done

  • Done is when people with motor, visual, or cognitive disabilities can exercise all four freedoms on the feature
  • Include accessibility in acceptance criteria and PR templates
  • Ship with automated a11y checks and human review
Reinforce: code merge is not “done” until POUR is satisfied for real users. Mirror security practice: automate checks, then verify manually. Treat accessibility breakage as a release blocker.
